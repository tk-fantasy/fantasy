# 摄像头接入与配置

Aether 通过一个本地摄像头「看」家里发生了什么，再把画面交给视觉大模型（VL）理解。这一篇讲怎么把摄像头接进来、参数怎么调、面板怎么用。

## 一、它整体怎么工作

```
摄像头 ──抓帧──▶ 运动检测(dHash) ──有变化──▶ 视觉模型(VL) ──▶ 识别结果
                                          │
                                          ├─▶ 摄像头面板显示（/camera）
                                          └─▶ 触发自动化规则评估
```

两个关键设计，先记住，后面会展开：

- **运动门控**：不是每一帧都调 VL（贵且慢），而是先用一个极轻量的 dHash 比对，画面没动就跳过。只有检测到运动才触发推理。
- **关注项**：你可以告诉模型「重点看什么」（比如「猫有没有上沙发」），它识别时就会带上有针对性的问题。详见《视觉关注项配置》。

## 二、硬件与接入

### 1. USB 摄像头（无 RTSP 时的默认）

最简单的方式。把 USB 摄像头插到运行 Aether 的电脑上，配置里 `camera_index` 填设备号：

> 接入优先级：填了 `rtsp_url` 走网络摄像头（见下一节），没填才回落 USB。两种二选一，不用同时配。

```json
"vision": {
  "camera_index": 0
}
```

- `0` 通常是第一个摄像头；如果有多个，依次试 `1`、`2`。
- 笔记本自带的摄像头一般也是 `0`。

### 2. 网络摄像头（RTSP）

家里有 IP 摄像头（海康、大华、小米小默等支持 RTSP 的），可以走网络流。RTSP 优先级高于 USB——填了 `rtsp_url` 就用网络摄像头，留空才回落 USB。

最方便的配置方式是**在「高级配置」页（`/advanced`）的「视觉参数」段填表**：

- **摄像头源（RTSP）**：填完整地址，如 `rtsp://192.168.1.100:554/stream`
- **RTSP 用户名**：摄像头登录用户名（无鉴权留空）
- **RTSP 密码**：留空表示不修改，填了会加密存进 `.env`（不进 config.json）

也可以直接编辑配置文件。`config.json` 的 `vision` 段：

```json
"vision": {
  "rtsp_url": "rtsp://192.168.1.100:554/stream2",
  "rtsp_username": "admin",
  "rtsp_password_env": "RTSP_PASSWORD"
}
```

密码**不写明文**：`rtsp_password_env` 填的是 `.env` 里的变量名（如 `RTSP_PASSWORD`），真正的密码值放在 `.env`：

```
RTSP_PASSWORD=your_camera_password
```

后端连接时把 `rtsp://host/path` 拼成 `rtsp://user:pwd@host/path`，用 `cv2.CAP_FFMPEG` 打开。没配凭证则裸连（支持无鉴权摄像头）。改完触发摄像头重连即生效（worker 会自动重连，无需重启服务）。

> RTSP 走的是 FFmpeg 后端，底层是阻塞 IO，后端在独立线程里抓帧，不会卡住主事件循环。

### 3. 带 PTZ（云台）的网络摄像头

如果你的摄像头支持 ONVIF 协议且能转动，可以开启 PTZ，在聊天里输入 `/camera` 打开面板后用方向键控制转动。配置项在 `ptz` 段：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `ptz.enabled` | `false` | 是否启用云台。关掉则面板不显示方向键 |
| `ptz.ip` | `""` | 摄像头 IP |
| `ptz.port` | `80` | ONVIF 端口 |
| `ptz.username` | `""` | 登录用户名 |
| `ptz.password_env` | `""` | 密码从环境变量读取（填变量名，不填明文） |
| `ptz.step_ms` | `300` | 点一下方向键转动的毫秒数，也用作前端点击冷却 |
| `ptz.speed` | `0.5` | 转动速度，`0.1~1.0` |

> PTZ 用的是 ONVIF 协议，底层是同步阻塞的 zeep client，后端用 `asyncio.to_thread` 丢进线程池，避免慢握手卡住视频流。

## 三、视觉参数详解

`vision` 段是视觉部分的核心配置。下面这张表是 `config.example.json` 里的默认值，逐项说明：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `camera_index` | `0` | 摄像头设备号 |
| `downscale_max_side` | `448` | 抓帧后缩放的最大边长（像素）。越小越省显存和 token，但细节也越少 |
| `jpeg_quality` | `70` | 编码成 base64 时的 JPEG 质量。`70` 是清晰度和体积的平衡点 |
| `motion_hash_size` | `16` | dHash 哈希尺寸。`16` → 256 位指纹 |
| `motion_threshold` | `15` | 运动距离阈值，超过才触发推理。**调大 = 更迟钝（只对大动作反应），调小 = 更灵敏** |
| `motion_check_interval_seconds` | `0.2` | 多久做一次运动检测。`0.2` 即每秒 5 次 |
| `min_infer_interval_seconds` | `3.0` | 两次推理之间的最小间隔，避免狂调模型 |
| `max_idle_interval_seconds` | `60.0` | 画面一直没动时，最长多久强制推一次（确认场景还在） |
| `vision_use_img_count` | `3` | 一次推理喂给模型几帧。多帧能看出动作趋势，但更费 token |
| `frame_interval_ms` | `1000` | 抓帧间隔，`1000` 即每秒一帧 |
| `read_retry_count` | `3` | 读帧失败重试次数 |
| `read_retry_interval_seconds` | `0.1` | 重试间隔 |
| `release_cooldown_seconds` | `0.8` | 摄像头释放后的冷却时间，避免频繁抢占 |
| `max_backoff_seconds` | `15.0` | 设备掉线后重连的指数退避上限 |
| `rtsp_url` | `""` | RTSP 网络摄像头地址。**填了走网络流，留空走 USB**。如 `rtsp://192.168.1.100:554/stream` |
| `rtsp_username` | `""` | RTSP 用户名（无鉴权摄像头留空） |
| `rtsp_password_env` | `""` | RTSP 密码的环境变量名（填变量名不填明文，真密码放 `.env`）。前端配置页填密码时后端自动设为 `RTSP_PASSWORD` |
| `slow_read_ms_threshold` | `500.0` | 读帧耗时超过此毫秒数计入慢读计数 |
| `slow_read_reopen_threshold` | `3` | 连续慢读次数超此值则重开设备（USB 卡顿自愈） |
| `backend_blacklist_seconds` | `30.0` | 慢读拉黑 backend 的时长，TTL 内跳过该 backend |
| `infer_timeout_seconds` | `45.0` | 单次视觉推理超时 |

> RTSP 密码为什么这么绕？——密码是敏感信息，明文写进 `config.json` 会跟着 git 泄露。`rtsp_password_env` 只存变量名，真密码留在 `.gitignore` 忽略的 `.env` 里，跟 LLM Key 的处理方式一致。前端「高级配置」页填密码时会自动走这套流程，用户不用手动管变量名。

### 常见调整场景

- **太灵敏，没人也一直触发** → 调大 `motion_threshold`（如 `20`、`25`）。
- **有人进来却没反应** → 调小 `motion_threshold`（如 `10`），或检查 `downscale_max_side` 是否太小导致人影糊掉。
- **想省 token / 模型慢** → 调大 `min_infer_interval_seconds`，或把 `vision_use_img_count` 改成 `1`。
- **夜间灯光闪烁误触发** → 调大 `motion_threshold`，或调大 `motion_check_interval_seconds`。

## 四、摄像头面板（/camera）

在聊天框输入 `/camera` 打开摄像头面板。面板从上到下是：

1. **视频预览**：实时 MJPEG 画面。右上角徽标显示 `已连接` / `等待连接`。
2. **状态条**：画面断了会显示「设备重连中」并提供「重试」按钮。前端最多重连 10 次（指数退避，约 5 分钟），期间只要设备恢复就自动接回。
3. **云台方向键**（仅 `ptz.enabled=true` 时显示）：点一下转一小段（`ptz.step_ms` 毫秒）后自动停。
4. **统计区**：
   - `运动距离`：当前 dHash 距离，超过 `motion_threshold` 才触发推理。
   - `推理次数`：累计调用 VL 的次数。
   - `模型 FPS`：推理频率上限（≈ `1 / min_infer_interval`）。
5. **识别反馈**：模型当前对画面的描述，比如「沙发上有人」。

> 面板里的「正在确认…」是显示层平滑：连续几帧都看到事件才正式显示结论，避免单帧误判闪一下。这只是给看的，不影响自动化规则触发（规则直接用模型判断结果，不走这个平滑）。详见《运动检测与视觉推理》。

## 五、相关接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/video_feed` | GET | MJPEG 视频流（`multipart/x-mixed-replace`），需 JWT 认证 |
| `/api/state` | GET | 全局状态，含 `camera_opened`、`motion_distance`、`infer_count`、`model_fps`、`feedback` 等。面板每 2 秒拉一次 |
| `/api/ptz/status` | GET | 云台是否启用 + 单步时长 |
| `/api/ptz/move` | POST | 按住式持续转动（方向 `up/down/left/right`） |
| `/api/ptz/stop` | POST | 停止转动 |
| `/api/ptz/step` | POST | 点按式步进，转一小段自动停 |

## 六、不开视觉模型会怎样

如果没配置 `vision` 角色的 LLM Key（见《LLM模型角色配置》），视觉模型处于「未启用」状态。这时：

- 摄像头照常抓帧、面板照常显示画面。
- 但不会有识别结果，面板反馈显示「视觉模型未启用，当前只有摄像头画面，没有识别。」
- 自动化里的「视觉触发规则」退化为纯上下文规则（只看时间/天气，不看画面）。

所以要完整使用视觉能力，务必配好 `vision` 角色的 Key。

## 七、排查

| 现象 | 排查方向 |
|------|----------|
| 面板一直「等待连接」 | USB：检查 `camera_index` 是否正确、是否被别的程序占用、看后端日志有无读帧报错；RTSP：确认 `rtsp_url` 能通（VLC 试播）、用户名密码对不对、摄像头是否开了 RTSP 服务 |
| 画面有但从不识别 | 确认 `vision` 角色 Key 已配；看 `motion_threshold` 是否过高；确认画面确实有运动 |
| 识别太频繁/太慢 | 调 `min_infer_interval_seconds` 和 `motion_threshold` |
| PTZ 方向键不显示 | 确认 `ptz.enabled=true` 且 `ptz.ip` 等填对；面板打开时会拉一次 `/api/ptz/status` |
| 设备掉线后不恢复 | 看 `max_backoff_seconds` 和重试次数；USB 拔插后可能需要重启服务；RTSP 网络抖动会自动指数退避重连 |
| USB 画面卡顿后自愈 | 后端有慢读检测：连续 `slow_read_reopen_threshold` 次读帧超 `slow_read_ms_threshold` 毫秒，会自动重开设备并临时拉黑该 backend `backend_blacklist_seconds` 秒 |
