# 运动检测与视觉推理

这一篇讲摄像头视觉的内部机制：画面是怎么一步步变成识别结果和自动化触发的。偏技术，配过摄像头、想理解「为什么有时不识别」「为什么有延迟」的读者可以看。

## 一、整体流水线

```
抓帧(frame_interval_ms=1000)
   │
   ▼
运动检测(dHash, 每 motion_check_interval_seconds=0.2 一次)
   │  比对当前帧 vs 参考帧(上一次真正推理的那帧)
   │
   ├─ 距离 ≤ motion_threshold(15) ──▶ 视为静止，跳过
   │                                    │
   │                                    └─▶ 但若距上次推理 > max_idle_interval(60s)
   │                                        强制推一次（确认场景还在）
   │
   └─ 距离 > threshold ──▶ 触发推理
                              │
                              ├─ 距上次推理 < min_infer_interval(3s) ? 跳过（节流）
                              │
                              ▼
                   取最近 vision_use_img_count(3) 帧打包
                              │
                              ▼
                   VL 模型推理(classify_frame)
                              │
                              ├─▶ 写回 CameraState（面板显示）
                              └─▶ 回调 on_inference_done → 触发自动化规则评估
                              │
                              ▼
                   commit_reference()：把这一帧设为新参考帧
```

## 二、运动门控：dHash 为什么比「上一帧」聪明

运动检测用的是 **dHash（差值感知哈希）**：把帧转灰度、缩到 `17×16`、比较相邻像素大小，得到一个 `256` 位的指纹。两帧的汉明距离越大，画面差异越大。代价只有一次缩放，微秒级。

关键设计在**参考帧的选择**：

> 比较对象是「上一次送模型推理的那一帧」，而不是「上一帧」。

为什么这么设计？看两种情况的对比：

| 场景 | 比「上一帧」 | 比「上一次推理帧」（实际做法） |
|------|--------------|--------------------------------|
| 人缓慢走入画面 | 每两帧差异都很小，可能永远触发不了 | 差异相对参考帧持续累积，到阈值就触发 |
| 推理后画面恢复平静 | 每帧都和上一帧比，没事 | 和推理时的帧比，确认恢复平静 |

实现上：

- `assess(frame)` 算当前帧哈希，和 `_reference_hash` 比距离，同时存到 `_last_hash`。
- `commit_reference()` 在**推理真正发生后**调用，把 `_last_hash` 提升为 `_reference_hash`。

也就是说，参考帧只在「真正调过模型」之后才更新。两次推理之间的所有抓帧，都拿同一帧做参照，差异是累积的。

## 三、推理节流：三道闸门

为了避免狂调模型（又贵又慢），推理前面有三道闸门：

1. **运动闸门**：`motion_distance > motion_threshold(15)` 才放行。`15` 是 `16×16` 哈希下的经验值，对应大约 6% 的像素发生变化。
2. **最小间隔闸门**：距上次推理不足 `min_infer_interval_seconds(3.0)` 就跳过。即使画面一直在动，最多每 3 秒推一次。
3. **空闲兜底闸门**：如果一直没运动，超过 `max_idle_interval_seconds(60.0)` 强制推一次。防止场景已经变了但 dHash 没察觉（比如光照渐变）。

三道闸门是「或」关系：运动触发 ∨ 空闲超时，再叠加最小间隔节流。

## 四、多帧推理：为什么喂 3 帧而不是 1 帧

`vision_use_img_count=3` 表示一次推理把最近 3 帧打包发给 VL 模型。好处是模型能看到**动作趋势**（比如「人正在站起来」vs「人坐着」），判断更准。代价是 token 更多。

帧间隔由 `frame_interval_ms=1000` 决定，所以 3 帧覆盖约 2 秒的窗口。

## 五、跨线程调度：run_coroutine_threadsafe

摄像头抓帧和推理跑在一个独立线程里（OpenCV 的 `cv2.VideoCapture` 是同步阻塞的，不能放在 asyncio 事件循环里）。但 VL 推理是 `async` 的（用 httpx 连接池）。怎么衔接？

```
摄像头线程(同步) ──run_coroutine_threadsafe──▶ 主事件循环(asyncio) ──▶ VL 推理
```

`run_coroutine_threadsafe(coro, loop)` 把一个协程投递到主事件循环里执行，摄像头线程不阻塞。事件循环在服务启动时通过 `set_event_loop` 注入到 `CameraStream`。

推理完成后，通过 `call_soon_threadsafe` 调度 `on_inference_done` 回调，触发自动化规则评估。这条路径是**事件驱动**的——有识别结果才评估，不等 10 秒定时 tick。

## 六、显示平滑：presence_threshold 是给看的，不是给规则用的

摄像头面板上你会看到「检测到事件，正在确认…」这种中间态。这是 `_resolve_display_result` 做的平滑：

- 每次推理有事件（`action` 不是 `idle`/`no_event`）→ `_presence_count++`。
- 连续 `_presence_threshold(3)` 帧有事件 → 正式显示结论（`confirmed=true`）。
- 连续 3 帧无事件 → 显示「画面平静」。
- 中间状态 → 显示「正在确认…」。

**这个平滑只影响面板显示，不影响自动化规则。** 自动化评估走的是另一条路：

```
自动化 tick(每 10s) / 事件回调
   │
   ▼
vision_service.evaluate_condition(frames, 条件, context)  ← 直接调 VL，返回 0/1
   │
   ▼
满足条件 → 执行规则动作
```

`evaluate_condition` 单独调一次 VL，让它针对规则的**自然语言条件**判断 `0`（不成立）或 `1`（成立），不经过 `_presence_count` 那套平滑。所以：

- 面板上「正在确认」时，规则可能已经触发了。
- 面板上「确认」了，不代表规则一定触发（规则有自己的条件判断）。

两套机制各司其职：显示平滑为了不闪屏，规则判断为了准和快。

## 七、一次 tick 编码一次：automation 的优化

自动化一个 tick（10 秒）可能要评估多条视觉规则。如果每条规则都把 3 帧重新编码成 base64，是浪费。所以：

```
automation_service.evaluate(frames)
   │
   ├─ encode_frames_b64(frames)  ← 一次
   │
   └─ 对每条规则:
        vision_service.evaluate_condition(frames, 条件, pre_encoded_b64=上面那份)
```

`pre_encoded_b64` 复用同一份编码，5 条规则也只编一次。编码失败则降级为每条规则各自编码（容错）。

## 八、没有摄像头画面时

如果摄像头没接好或视觉服务没启用，自动化 tick 拿不到 `frames`。这时：

- 视觉规则**不会跳过**，而是走**纯上下文模式**（`use_context_only`）。
- 只用时间、天气等上下文评估条件，视觉条件默认不成立。
- 日志会打 `No camera frames, evaluating N rules with context`。

所以「摄像头掉了」不会让所有规则失效——纯时间/天气条件照常工作，只是视觉条件无法满足。

## 九、参数速查

| 参数 | 默认 | 作用 | 调整方向 |
|------|------|------|----------|
| `motion_threshold` | `15` | 运动触发阈值 | 大→迟钝；小→灵敏 |
| `motion_hash_size` | `16` | dHash 尺寸 | 一般不动 |
| `motion_check_interval_seconds` | `0.2` | 运动检测频率 | 一般不动 |
| `min_infer_interval_seconds` | `3.0` | 推理最小间隔 | 大→省 token；小→更及时 |
| `max_idle_interval_seconds` | `60.0` | 空闲强制推理 | 大→更省；小→更勤确认 |
| `vision_use_img_count` | `3` | 单次推理帧数 | 1→省 token；3→看趋势 |
| `frame_interval_ms` | `1000` | 抓帧间隔 | 小→更密；大→更省 |
| `downscale_max_side` | `448` | 缩放最大边 | 小→省 token；大→更清晰 |
| `jpeg_quality` | `70` | JPEG 质量 | 一般不动 |

## 十、常见疑问

**Q：画面里明显有人，为什么不识别？**
A：依次查：① 视觉模型 Key 配了吗（`vision` 角色）；② `motion_threshold` 是否过高；③ `downscale_max_side` 是否太小导致人影糊掉；④ 距上次推理是否在 `min_infer_interval` 内被节流。

**Q：面板显示「正在确认…」但规则已经动作了，正常吗？**
A：正常。显示有平滑（3 帧确认），规则判断是独立的。见第六节。

**Q：人静止不动很久，会一直触发推理吗？**
A：不会。人静止后 dHash 距离回到阈值以下，停止触发。但每 `max_idle_interval(60s)` 会强制推一次确认场景。

**Q：缓慢走入画面能识别吗？**
A：能。因为参考帧是「上次推理的帧」而不是「上一帧」，缓慢入画的差异会累积到阈值。见第二节。
