# 系统健康检查指南

想让 Aether 稳稳跑着，偶尔「体检」一下很有必要。这篇讲怎么看系统状态、怎么快速定位问题。

## 一、总体健康检查

浏览器打开（需先登录）：

```
http://localhost:8010/api/health
```

返回字段：

| 字段 | 含义 |
|------|------|
| `status` | `ok` 表示服务在跑 |
| `llm_model` | 当前用的聊天模型 |
| `llm_enabled` | LLM 是否启用 |
| `llm_available` | LLM 是否真的能调通（启动时后台探一次） |
| `ha_available` | Home Assistant 是否连得上 |
| `camera` | 摄像头完整状态（见下） |
| `log_file` | 日志文件路径（`logs/app.log`） |

**`status: ok` + `llm_available: true` + `ha_available: true` = 一切正常。**

> 注意：这个接口要登录才能访问。想做无认证的健康探针（Docker/k8s），目前没有现成的。

## 二、组件状态检查

### 自动化规则在不在跑

```
http://localhost:8010/api/agents/status
```

返回：

| 字段 | 含义 |
|------|------|
| `status` | `not_started` 表示自动化没启动 |
| `automation.running` | 是否在运行 |
| `automation.eval_interval` | 评估周期（默认 10 秒） |
| `automation.eval_count` | 累计评估次数 |

> 这个接口**没有**调度器（scheduler）字段。定时任务的运行状态看 `/api/scheduled-tasks` 里每条任务的 `last_status`（`running`/`success`/`failed`/`interrupted`）。

### 摄像头状态

```
http://localhost:8010/api/state
```

关键字段：

| 字段 | 含义 |
|------|------|
| `camera_opened` | 摄像头有没有成功打开 |
| `frame_width` / `frame_height` | 画面分辨率 |
| `fps` / `worker_fps` | 抓帧 / 处理帧率 |
| `motion_distance` | 当前运动距离（超 `motion_threshold` 才触发推理，默认 -1 表示无参考帧） |
| `motion_threshold` | 运动阈值（默认 5，config 里一般设 15） |
| `infer_count` | 累计推理次数 |
| `model_fps` | 推理频率上限 |
| `infer_busy` | 是否正在推理 |
| `feedback` | 当前识别反馈文本 |
| `last_error` | 最近一次错误 |

### 指标快照

```
http://localhost:8010/api/metrics
```

返回请求计数/延迟、工具调用次数、LLM 调用次数等，用于看负载情况。

## 三、排查速查表

按这个顺序过，大多数问题能定位：

| 步骤 | 检查什么 | 怎么做 | 正常应该是 |
|------|----------|--------|-----------|
| 1 | Docker 容器 | `docker compose ps` | mqtt + homeassistant 都是 `Up` |
| 2 | 后端服务 | `http://localhost:8010/api/health` | `status: ok` |
| 3 | 前端页面 | `http://localhost:5173`（开发）或 `http://localhost:8010`（生产） | 显示登录页 |
| 4 | HA 连接 | health 里的 `ha_available` | `true` |
| 5 | LLM 可用 | health 里的 `llm_available` + `/keys` 测试 | `true` |
| 6 | 摄像头 | `/api/state` 的 `camera_opened` 或 `/camera` 面板 | `true`，有画面 |
| 7 | 自动化 | `/api/agents/status` 的 `running` | `true` |

哪步不对，从那步开始查。

## 四、Docker 服务状态

Aether 依赖 **2 个** Docker 服务（不是 3 个，搜索已改用云端 Exa，不再有 SearXNG 容器）：

| 服务 | 容器名 | 端口 | 作用 |
|------|--------|------|------|
| mqtt | `mosquitto` | 1884 | MQTT 消息代理 |
| homeassistant | `aether-ha` | 8123 | 智能家居平台 |

```powershell
# 查状态
docker compose ps

# 看日志（最近 50 行）
docker compose logs homeassistant --tail 50
docker compose logs mqtt --tail 50
```

> HA 用的是本地镜像 `aether-ha:local`，不是 ghcr.io 官方镜像。第一次要用配套脚本构建。

## 五、常见情况解读

### 摄像头没打开

- USB 没插好 / 被会议软件占用 / 笔记本物理开关关了
- `camera_index` 不对，试 `0`/`1`/`2`
- 看日志 `Get-Content logs\app.log | Select-String "camera"`

### 运动距离一直很高

- 画面里确实有动（风扇、窗帘飘）
- 静止时还高 → 摄像头噪点，调大 `motion_threshold`

### 视觉识别卡住

- LLM 太慢，系统有超时保护会自动恢复
- 经常发生 → 换快点的视觉模型，或调小 `downscale_max_side`

### `ha_available: false`

- HA 容器没跑：`docker compose ps homeassistant`
- HA 地址/Token 不对：检查 `config.json` 的 `ha` 段
- HA 还在启动中：等几十秒再查

### `llm_available: false`

- Key 没配或配错：`/keys` 页面检查
- Key 失效/额度用完：去提供商后台看
- 网络不通模型 API

### 自动化 `eval_count` 不增长

- 自动化没启动（`running: false`）
- 看日志有没有 `automation` 相关报错

## 六、测试入口

- **HA 连接测试**：`/settings` 或 `/ha` 页面的「测试」按钮
- **LLM Key 测试**：`/keys` 页面每条 Key 的「测试」按钮
- **设备控制测试**：`/halist` 手动操作设备，看是否响应

## 七、小贴士

- 先看 `/api/health` 总体，再看具体组件
- 按速查表 1→7 过一遍，通常能快速定位
- 还是不行 → `logs/app.log` 找 `ERROR`/`Exception`
- 大部分疑难杂症重启后端能解决
- DEBUG 模式（`LOG_LEVEL=DEBUG`）能看工具调用和规则评估细节，排完记得改回 `INFO`
