# 日志查看与问题排查

系统出问题时，日志是最好的朋友。这篇讲日志在哪、怎么看，以及常见问题的排查路径。

## 一、日志在哪

后端只写一个日志文件，带滚动归档：

| 日志 | 路径 | 谁写的 | 说明 |
|------|------|--------|------|
| 应用日志 | `logs/app.log` | 后端 Python | **主日志**，所有模块输出都在这 |
| 模拟器日志 | `logs/ha_simulator.log` | `ha_simulator.py` 重定向 | 设备模拟器的 stdout，`docker compose up` 启动才有 |
| 容器日志 | — | Docker | `docker logs aether` 查看后端容器 stdout |

> `ha_simulator.log` 不是后端主动写的，是设备模拟器进程把 stdout 重定向到文件。容器模式下日志目录通过 volume 挂载到宿主 `logs/`。

### 滚动归档

`logs/app.log` 用 `RotatingFileHandler`：单文件 10MB，保留 5 个备份（`app.log.1` ~ `app.log.5`），UTF-8 编码。不用担心日志撑爆磁盘。

## 二、日志级别

通过 `LOG_LEVEL` 环境变量控制（覆盖 `config.json` 的 `logging.level`）：

| 级别 | 适合 |
|------|------|
| `ERROR` | 只看错误 |
| `WARNING` | 错误 + 警告 |
| `INFO` | **默认**，能看到正常流程 |
| `DEBUG` | 排查疑难问题，会打印工具调用参数/返回、规则评估过程、摄像头细节 |

在 `.env` 里设 `LOG_LEVEL=DEBUG` 重启即可。日常用 `INFO` 就够，DEBUG 量很大。

## 三、常用日志命令

```powershell
# 实时看应用日志（最常用）
Get-Content logs\app.log -Wait -Tail 30

# 看最近的错误
Get-Content logs\app.log | Select-String "ERROR|Exception" | Select-Object -Last 20

# 看摄像头相关
Get-Content logs\app.log | Select-String "camera|vision|motion" | Select-Object -Last 20

# Docker 容器日志（最近 50 行）
docker compose logs homeassistant --tail 50
docker compose logs mqtt --tail 50
```

## 四、常见问题排查

### 前端页面打不开

1. **后端在不在**：`netstat -ano | findstr ":8010"`，没输出就重启后端。
2. **开发模式**：`http://localhost:5173`（Vite），看启动终端有没有报错。
3. **生产模式**：前端由 8010 端口统一提供，直接访问 `http://localhost:8010`。

### AI 不回消息

1. **密钥配了没**：`/keys` 页面看 `chat` 角色有没有绑 Key。
2. **密钥还能用没**：页面上点「测试」。
3. **额度用完没**：去 API 提供商后台看。
4. **网络通不通**：确认能访问模型 API 地址；用代理的话检查环境变量。
5. **看日志**：`Get-Content logs\app.log | Select-String "llm|chat|dispatch" | Select-Object -Last 20`。

### 摄像头打不开

1. **物理**：USB 插好没？被别的程序占着没（关掉会议软件）？笔记本快捷键关了没？
2. **看日志**：`Get-Content logs\app.log | Select-String "camera|VideoCapture" | Select-Object -Last 20`。
3. **`camera_index` 对不对**：多摄像头时试 `0`/`1`/`2`。

### 设备列表是空的

1. **HA 容器在跑没**：`docker compose ps homeassistant`，状态应 `Up`。
2. **模拟器跑了没**（demo 设备）：看 `logs/ha_simulator.log` 或重启模拟器——MQTT 设备状态不持久，HA 重启后要重新跑模拟器发布状态。
3. **看日志**：`Get-Content logs\app.log | Select-String "ha|entity" | Select-Object -Last 20`。

### 自动化规则不触发

1. **规则开了没**：`/task` 页面看开关。
2. **冷却期**：每条规则执行后有 10 秒冷却，期间不重复触发。
3. **视觉条件**：摄像头正常没？视觉模型 Key 配了没？没画面时视觉规则退化为纯上下文模式（只看时间/天气）。
4. **看日志**：`Get-Content logs\app.log | Select-String "automation|rule|eval" | Select-Object -Last 20`。

### 定时任务不执行

1. **任务开了没**：`/scheduled` 页面看开关。`at` 类型任务跑一次后自动禁用，这是正常的。
2. **时间解析对不对**：创建时看「调度摘要」是否符合预期。
3. **看日志**：`Get-Content logs\app.log | Select-String "scheduler|scheduled" | Select-Object -Last 20`。

### 搜索没结果（Exa）

1. **匿名限速了没**：留空 Key 匿名调用有速率限制，等一会儿或填自己的 Exa Key（`/advanced` 页面）。
2. **网络通不通**：要能访问 `https://mcp.exa.ai`。
3. **看日志**：`Get-Content logs\app.log | Select-String "web_search|exa" | Select-Object -Last 20`。

> 注：SearXNG 已移除，不要再找 searxng 容器或 8888 端口。

### 天气不对

- 天气 API 凭证配了没（`/keys` 或 `/advanced`）。
- 天气查询地是全局的，读 `config.json` 的 `home` 段（不跟用户走，一家人共用）。设置页改省市区不影响天气，要改天气地点得改 `config.json` 的 `home` 段再重启。

### 内存/CPU 占用高

- 视觉推理太频繁 → 调大 `min_infer_interval_seconds`（默认 3 秒）。
- 图片太大 → 调小 `downscale_max_side`（默认 448）。
- LLM 并发太高 → 调 `max_concurrency`。

## 五、DEBUG 模式

疑难问题开 DEBUG：

```
# .env
LOG_LEVEL=DEBUG
```

重启后能看到：
- 每个工具调用的参数和返回值
- 规则评估的中间过程（条件判断、是否触发）
- 摄像头抓帧、运动检测、推理的详细流水
- LLM 的请求/响应摘要

排完记得改回 `INFO`，DEBUG 日志量很大。

## 六、两个特殊端口

- **8011**：冷启动期间临时跑的「启动进度服务」，告诉你后端正在初始化。主服务（8010）起来后就关了。看到 8011 在监听是正常的。
- **5173**：仅开发模式（`npm run dev`）才有，生产环境不存在。

> 遇到报错别紧张，大部分问题看 `logs/app.log` 就能定位，实在不行重启后端往往就好。
