# 视觉子系统详解

> 对应代码：`app/camera_stream.py`、`app/services/motion_service.py`、`app/services/vision_service.py`、`app/services/priority_service.py`、`app/vision.py`

Aether 的视觉子系统实现摄像头采集、运动检测、VL 模型推理和状态发布的全链路。

---

## 1. 总体架构

```
┌──────────────────────────────────────────────────────────────┐
│ CameraStream (app/camera_stream.py)                          │
│                                                              │
│  ┌─────────────────────┐   ┌──────────────────────┐        │
│  │ _worker             │   │ _buffer_worker        │        │
│  │ (主采集线程, ~24fps) │   │ (帧缓冲线程)          │        │
│  │                     │   │                      │        │
│  │ opencv → JPEG编码   │   │ queue → frame_buffer  │        │
│  │ → MJPEG流 → 浏览器   │   │ (环形缓冲区)          │        │
│  └────────┬────────────┘   └──────────────────────┘        │
│           │                                                  │
│    _maybe_schedule_inference()                               │
│           │                                                  │
│     ┌─────▼──────┐    ┌────────────────────────┐            │
│     │ MotionDetector│  │ interactive_priority    │            │
│     │ dHash 检测   │  │ (用户聊天优先)          │            │
│     └─────┬──────┘    └────────────────────────┘            │
│           │                                                  │
│     ┌─────▼────────────────────────────────────┐             │
│     │ _infer_queue (queue.Queue, maxsize=5)     │             │
│     └─────┬────────────────────────────────────┘             │
│           │                                                  │
│     ┌─────▼────────────────────────────────────┐             │
│     │ _infer_scheduler_worker                   │             │
│     │ run_coroutine_threadsafe(_run_inference_  │             │
│     │   async, main_loop)  ← 跨线程投到主事件循环 │             │
│     └─────┬────────────────────────────────────┘             │
│           │ (主事件循环上)                                    │
│     ┌─────▼────────────────────────────────────┐             │
│     │ VisionService.classify_frame_async()      │             │
│     │ → VL Model API call (encode_frames_b64    │             │
│     │   用 downscale_max_side=448 降采样)        │             │
│     └─────┬────────────────────────────────────┘             │
│           │                                                  │
│     on_inference_done callback                               │
│       → AutomationAgent.trigger_evaluate()                   │
│         (call_soon_threadsafe 跨线程投回主循环)                │
└──────────────────────────────────────────────────────────────┘
```

> **跨线程关键**：摄像头采集/推理调度在独立线程，VL 推理协程通过 `run_coroutine_threadsafe` 投到主事件循环执行（与 10s tick 的 `evaluate()` 走相同的释放-GIL 路径）。推理完成后的 `on_inference_done` 回调再通过 `AutomationAgent.trigger_evaluate()` 的 `call_soon_threadsafe` 把规则评估投回主循环。两次跨线程投递，确保所有协程都在主事件循环上跑。

## 2. CameraStream 类

`app/camera_stream.py`

### 2.1 线程架构

CameraStream 运行 **4 个线程**：

| 线程 | 函数 | 职责 |
|------|------|------|
| 主采集线程 | `_worker` | 从摄像头读取帧、MJPEG 编码、运动检测调度 |
| 帧缓冲线程 | `_buffer_worker` | 从队列异步写入环形缓冲区 |
| 推理调度线程 | `_infer_scheduler_worker` | 从队列取帧、`run_coroutine_threadsafe` 投递推理协程到主循环 |
| 推理执行 | 主事件循环（`_run_inference_async`） | 串行执行 VL 模型推理；无主循环时回退 `ThreadPoolExecutor` |

> MJPEG 生成器 (`mjpeg_generator()`) 是一个同步生成器，由 FastAPI 的 `StreamingResponse` 协程模式驱动，不是专用线程。

### 2.2 摄像头上电策略

```python
def _open_camera(self) -> cv2.VideoCapture:
    candidates = [
        ("dshow:0", 0),   # Windows DirectShow 后端
        ("dshow:1", 1),
    ]
    for backend_name, camera_index in candidates:
        cap = cv2.VideoCapture(camera_index, cv2.CAP_DSHOW)
        # 等待 0.3s，采样 12 帧验证有效性
        time.sleep(0.3)
        for _ in range(12):
            ok, frame = cap.read()
            if ok and frame is not None:
                return cap
        cap.release()
    # 全部失败 → 空 VideoCapture (后续 _worker 每 1s 重试)
```

### 2.3 主循环 (_worker)

```python
def _worker(self) -> None:
    while self._running:
        # 1. 确保摄像头已打开
        if self._cap is None or not self._cap.isOpened():
            self._cap = self._open_camera()

        # 2. 读取帧
        ok, frame = self._cap.read()

        # 3. 运动门控 → 决定是否推理
        self._maybe_schedule_inference(frame)

        # 4. 叠加反馈文字 → JPEG 编码
        display_result = self._resolve_display_result(result)  # 显示平滑
        display_frame = self._prepare_display_frame(frame)      # 亮度增强
        encoded, jpeg = cv2.imencode(".jpg", display_frame,
                                      [cv2.IMWRITE_JPEG_QUALITY, jpeg_quality])

        # 5. 更新状态、存入环形缓冲
        self._latest_frame = frame
        self._latest_jpeg = jpeg.tobytes()
        if should_push_to_buffer:
            self._buffer_queue.put_nowait(frame)

        time.sleep(1/24)  # ~24fps 限制
```

### 2.4 MJPEG 流

```python
def mjpeg_generator(self):
    boundary = b"--frame\r\nContent-Type: image/jpeg\r\n\r\n"
    last_jpeg = None
    while self._running:
        frame = self.get_jpeg()
        if frame is None:
            time.sleep(0.05)
            continue
        if frame != last_jpeg:  # 仅帧变化时发送
            yield boundary + frame + b"\r\n"
            last_jpeg = frame
            time.sleep(1/24)
```

端点：`GET /api/video_feed` → `StreamingResponse(media_type="multipart/x-mixed-replace")`，handler 内显式校验 JWT。

## 3. 运动检测 (MotionDetector)

`app/services/motion_service.py`

### 3.1 dHash 算法

```python
def compute_dhash(frame_bgr, hash_size=16):
    """差值感知哈希."""
    gray = cv2.cvtColor(frame_bgr, cv2.COLOR_BGR2GRAY)
    resized = cv2.resize(gray, (hash_size + 1, hash_size),
                          interpolation=cv2.INTER_AREA)
    # 比较相邻列差异 → hash_size × hash_size 布尔矩阵
    return resized[:, 1:] > resized[:, :-1]
```

### 3.2 参考帧策略

```python
class MotionDetector:
    def assess(self, frame_bgr):
        current = compute_dhash(frame_bgr, self._hash_size)
        distance = hamming_distance(current, self._reference_hash)
        return distance > self._threshold, distance

    def commit_reference(self):
        """推理发生后调用，把当前帧设为新参考帧."""
        self._reference_hash = self._last_hash
```

**关键设计**：参考帧是"上次推理时的帧"而非"上一帧"（`commit_reference` 在推理发生后才调用）。缓慢入画的物体（如一个人慢慢走进画面）累积差异后也能触发推理，而不是每帧都和上一帧比（那样静止后就不再触发）。

## 4. 运动门控逻辑

`app/camera_stream.py:_maybe_schedule_inference()`

### 4.1 决策流程

```
if not vision_model.enabled:
    return                          # 视觉模型未启用

if now - last_motion_check < 0.2s:
    return                          # 检查间隔不足

moved, distance = motion.assess(frame)

if infer_busy:
    if timeout: reset infer_busy    # 安全阀：推理挂死恢复
    return                          # 推理中，跳过

if interactive_priority.active():
    return                          # 用户聊天中，让路

since_last = now - last_model_run_at

if moved and since_last >= min_infer_interval (3s):
    trigger = "motion"              # 运动触发
elif since_last >= max_idle_interval (60s):
    trigger = "heartbeat"           # 心跳兜底
else:
    return

# 标记忙碌，提交推理
infer_busy = True
infer_queue.put_nowait((frame, trigger))
motion.commit_reference()
```

两个触发条件是**或**关系：运动触发（`moved` 且超过最短间隔）或心跳兜底（超过最大空闲间隔）。心跳保证即便画面静止，每 60s 也至少推理一次，让规则评估能持续拿到新结果。

### 4.2 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `motion_hash_size` | 16 | dHash 尺寸 (16→256 位) |
| `motion_threshold` | 15 | 触发阈值 (0-256) |
| `motion_check_interval_seconds` | 0.2 | 检查间隔 |
| `min_infer_interval_seconds` | 3.0 | 最短推理间隔 |
| `max_idle_interval_seconds` | 60.0 | 最大空闲间隔 (心跳) |
| `vision_use_img_count` | 3 | 单次推理送入 VL 的帧数 |
| `frame_interval_ms` | 1000 | 多帧采样间隔 |
| `downscale_max_side` | 448 | VL 编码时降采样最大边长（保持比例） |
| `jpeg_quality` | 70 | VL 编码 JPEG 质量 |

### 4.3 推理超时安全阀

```python

        self._infer_busy = False  # 强制重置
```

## 5. VisionService

`app/services/vision_service.py`

### 5.1 classify_frame

```python
def classify_frame(self, frame_bgr) -> ActionResult:
    payload = asyncio.run(self._client.classify_frame(frame_bgr, focus=combined_focus))
    # 期望 JSON: {"event": "person_waving", "observation": "有人挥手"}
    event = parsed.get("event", "no_event")
    observation = parsed.get("observation", "")
    if event == "no_event":
        return ActionResult("no_event", observation or "画面平静，暂无事件。")
    return ActionResult(event, observation or feedback_default(event))
```

`focus` 由 `_get_combined_focus` 拼接所有 enabled 关注项（OR 关系），无关注项时默认 `"画面中的人和他们的行为"`。

### 5.2 evaluate_condition（自动化规则评估入口）

```python
async def evaluate_condition(self, frames, condition, context_info="", pre_encoded_b64=None) -> int:
    content = await self._client.evaluate_condition(
        frames, condition, context_info=context_info, pre_encoded_b64=pre_encoded_b64,
    )
    # 容错：从响应中提取第一个数字
    match = re.search(r'\d+', content)
    if match:
        raw_num = int(match.group())
        return 0 if raw_num == 0 else 1     # 0=不成立，非0一律当成立
    return 0
```

**关键**：`evaluate_condition` 返回 0/1，由 `AutomationService` 调用，**与下面的 `presence_threshold` 显示平滑完全独立**——自动化规则直接用这个 0/1 判断是否触发动作，不经过显示状态机。

`pre_encoded_b64` 是 `AutomationService` 一个 tick 编码一次的复用 b64（见 5.4），传入则跳过内部编码。

### 5.3 Vision Focus 管理

```python
# 多条 CRUD
vision_service.add_focus("桌子上有没有快递")
```

关注项同时用于 `classify_frame`（事件分类提示词）和系统提示词注入。详见《05-摄像头视觉/视觉关注项配置》。

### 5.4 encode_frames_b64（一个 tick 编码一次复用）

```python
async def encode_frames_b64(self, frames: list) -> list[str]:
    """批量编码帧为 base64，供一个 tick 内多条规则复用同一份编码。"""
    return await asyncio.to_thread(
        _encode_frames_b64, frames, client._max_side, client._jpeg_quality
    )
```

`AutomationService.evaluate` 在视觉分支开头调一次，把结果作为 `pre_encoded_b64` 透传给每条规则的 `evaluate_condition`，避免 N 条规则各自重复 `imencode + base64`（N 条规则省 N-1 次编码）。编码用 `downscale_max_side=448` 降采样（保持比例）+ `jpeg_quality=70`，走 `asyncio.to_thread` 不在事件循环里持 GIL。

## 6. ActionResult 与显示状态机

`app/vision.py`

```python
@dataclass
class ActionResult:
    action: str        # 事件类型: "idle", "no_event", "person_detected", ...
    feedback: str      # 人类可读的描述
    details: dict      # 附加信息: {source, enabled, event, observation, model, ...}
```

### 显示平滑状态机（_resolve_display_result）— 仅影响视频流叠加文字

`CameraStream._resolve_display_result()` 对**视频流上显示给用户的事件**做去抖平滑，`_presence_threshold = 3`：

```python
has_event = result.action not in {"idle", "no_event"}
if has_event:
    self._presence_count += 1;  self._absence_count = 0
else:
    self._absence_count += 1;  self._presence_count = 0

if self._presence_count >= 3:        # 连续 3 帧有事件 → 确认显示
    return ActionResult(action=result.action, feedback=result.feedback, ...)
if self._absence_count >= 3:         # 连续 3 帧无事件 → 回到 idle
    return ActionResult(action="idle", feedback=...)
if has_event:                         # 有事件但不足 3 帧 → 等待确认
    return ActionResult(action="waiting_confirm", feedback="检测到事件，正在确认…", ...)
return ActionResult(action="idle", feedback=...)
```

> **⚠️ 关键区分：这是显示专用，不影响自动化触发。**
> - **显示路径**：`classify_frame` → `_resolve_display_result`（presence_threshold=3 平滑）→ 视频流叠加文字 / `/api/state` 的 `action` 字段。
> - **自动化路径**：`evaluate_condition` → 直接返回 0/1 → `AutomationService` 据此触发动作。
>
> 两条路径独立——自动化规则**不读** `presence_count`，也不经过 `waiting_confirm` 状态。`presence_threshold=3` 只是避免视频流上事件文字频繁闪烁的平滑措施。

## 7. 交互优先级系统

`app/services/priority_service.py`

```python
class InteractivePriority:
    """用户聊天时阻塞后台视觉推理，避免本地 LLM 串行队列阻塞。"""
    def acquire(self): self._count += 1
    def release(self): self._count = max(0, self._count - 1)
    def active(self): return self._count > 0

interactive_priority = InteractivePriority()
```

在 Dispatcher 处理用户请求时自动 acquire，完成后 release。`_maybe_schedule_inference()` 检测到 active 时跳过推理调度——让本地 LLM 的串行队列优先服务用户聊天，而非后台视觉推理。

## 8. 推理任务调度

### 8.1 队列与跨线程投递

```
采集线程  →  _infer_queue (queue.Queue, maxsize=5)
                │
                ▼
           _infer_scheduler_worker (独立线程)
                │
                ├─ 主循环可用 → run_coroutine_threadsafe(_run_inference_async, main_loop)
                │                （投到主事件循环跑，与 10s tick evaluate 同路径）
                │
                └─ 主循环不可用 → _infer_executor.submit(_run_inference)  [fallback]
```

主路径是 `run_coroutine_threadsafe`——把推理协程从摄像头调度线程投到主 asyncio 事件循环执行。`ThreadPoolExecutor`（`_infer_executor`）仅作为未注入事件循环时的回退。

### 8.2 推理执行与回调

```python
async def _run_inference_async(self, frame):
    result = await self._recognizer.classify_frame_async(frame)   # VL 模型调用
    with self._lock:
        self._latest_result = result
        self._infer_busy = False
    # 触发规则评估（跨线程投回主循环）
    if self._on_inference_done:
        self._on_inference_done()   # → AutomationAgent.trigger_evaluate()
```

`on_inference_done` 回调里 `AutomationAgent.trigger_evaluate()` 用 `call_soon_threadsafe` 把规则评估投回主循环——注意此时已经在主循环上下文，但 `trigger_evaluate` 的线程安全设计保证从任何线程调用都安全。

## 9. 配置调优建议

| 场景 | motion_threshold | min_infer_interval | max_idle_interval |
|------|:---:|:---:|:---:|
| 室内监控（低活动） | 12 | 5s | 120s |
| 室内监控（高活动） | 20 | 3s | 60s |
| 节能模式（本地模型） | 18 | 10s | 180s |
| 高响应模式（云端模型） | 10 | 1s | 30s |

> 调小 `motion_threshold` 更敏感（但易误触）；调大 `min_infer_interval` 省 token（但反应慢）；`max_idle_interval` 是心跳兜底，保证静止画面也能周期性评估规则。VL 推理的超时由底层 httpx 客户端控制（非独立配置项），自动化整轮评估超时见 `_EVAL_TIMEOUT_SECONDS=60`（《自动化引擎详解》）。
