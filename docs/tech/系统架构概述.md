# Aether 系统架构概述

## 1. 总体架构

Aether 是单体分层架构，运行时由一个 FastAPI 后端 + Vue3 前端 + 两个 Docker 容器（MQTT、Home Assistant）组成：

```
┌────────────────────────────────────────────────────────────────────┐
│ Browser (localhost:5173 dev / 8010 prod)                           │
│  Vue3 + Vite SPA    ←→  WebSocket  ←→  REST API                   │
└────────────────────────────────────────────────────────────────────┘
                              │  (JWT cookie 鉴权)
┌─────────────────────────────▼──────────────────────────────────────┐
│ Aether Backend (localhost:8010)           Python / FastAPI         │
│                                                                   │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────────────────┐   │
│  │ Dispatcher│→│ LangGraph     │→│ MCP Tool Layer             │   │
│  │ (WS/REST) │  │ ReAct Agent  │  │ (12 内置 + external stdio)│   │
│  └──────────┘  └──────────────┘  └───────────────────────────┘   │
│                                                                   │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │ Services: VisionService  AutomationService  RuleRegistry     │ │
│  │           SchedulerService  HAService  WeatherService        │ │
│  │           SessionStore  HealthChecker  EmojiService          │ │
│  │ CameraStream (capture + motion-gated VL inference)           │ │
│  └──────────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────────┘
         │                                       │
┌────────▼───────────────┐              ┌────────▼─────────┐
│ Home Assistant         │   MQTT       │ Mosquitto MQTT   │
│ (localhost:8123)       │ ←──────────→ │ (localhost:1884) │
│ image: aether-ha:local │              │ eclipse-mosquitto│
└────────────────────────┘              └──────────────────┘
```

> 搜索走云端 Exa MCP（`https://mcp.exa.ai/mcp`），不再有本地 SearXNG 容器。

## 2. 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | Vue 3 + Vite | SPA，dev 5173，prod 由 8010 统一提供 |
| 后端 | FastAPI (async) | Uvicorn，端口 8010 |
| AI 编排 | LangGraph | `create_react_agent`，ReAct 循环 |
| 工具协议 | MCP (Model Context Protocol) | 内置 12 工具 + 外部 stdio JSON-RPC |
| 视觉 | OpenCV (cv2) + VL LLM | dHash 运动门控，MJPEG 流 |
| 文本模型 | Chat LLM | glm-4-flash 等，按 `chat` 角色 |
| 嵌入模型 | embed LLM | Emoji 语义搜索、RAG 检索 |
| 运动检测 | dHash | NumPy 实现，微秒级 |
| 数据库 | SQLite (aiosqlite) | sessions / rules / scheduled_tasks / user_settings / emoji_preferences |
| 消息队列 | MQTT (Mosquitto) | 设备状态通信 |
| 搜索 | Exa | 云端 MCP 端点 |
| 鉴权 | JWT (HS256) | access 24h / refresh 7d，httpOnly cookie；APP_TOKEN 兜底 |
| 容器化 | Docker Compose | 2 个服务（mqtt + homeassistant） |

## 3. 依赖注入：AppContainer

所有服务集中在 `AppContainer`（`app/container.py`，`@dataclass`）里，通过 `Depends(get_container)` 注入到路由：

**字段分组**：

- **Clients**：`ha_service`、`llm_chat_client`、`vision_client`、`summary_client`、`embed_client`
- **Services**：`session_store`、`vision_service`、`vision_key_pool`、`rule_service`、`rule_registry_service`、`automation_service`、`summarization_service`、`llm_settings_service`、`emoji_service`、`metrics_service`
- **MCP/工具**：`mcp_client_manager`、`tool_executor`
- **Camera**：`camera_stream`
- **可变引用（热替换，`list[0]` 模式）**：`ha_client_ref`、`automation_agent_ref`、`ha_catalog_cache_ref`、`ha_controls_cache_ref`
- **延迟赋值（lifespan 里填，默认 `None`）**：`dispatcher`、`scheduler_service`、`rag_service`

`ha_client` 是个 property，动态返回 `ha_client_ref[0]`，支持 HA 客户端热替换。`reload_all_clients()` 在用户改 LLM Key 后重载 chat/vision/summary/embed 客户端。

## 4. 数据流

### 4.1 用户聊天（主流程）

```
User Message
    │
Frontend ──WebSocket /ws/chat?token=──→ FastAPI
    │                                       │
    │                                  Dispatcher.dispatch_stream(event, ws_send, user_id)
    │                                       │
    │                                  SessionStore.get_or_create(session_id, ..., user_id)  ← 会话按用户隔离
    │                                       │
    │                                  _prepare_context():
    │                                    - HA 设备目录 + 控件文本
    │                                    - 视觉关注项
    │                                    - 摘要刷新
    │                                    - build_system_prompt(persona + guidelines + 时间 + 设备 + 控件 + 天气 + 摘要)
    │                                    - session → LangChain messages
    │                                       │
    │                                  run_agent_streaming(agent, messages, session, timeout=120s)
    │                                    - agent.astream_events(version="v2")
    │                                    - on_chat_model_stream → token
    │                                    - on_tool_start / on_tool_end → 工具调用
    │                                       │
    │              Stream Events: UI.Status / Template.TokenStream /
    │              Template.CallTool / Template.CallToolResult / Dialog.Finish
    │                                       │
    ◄────────────── WebSocket ──────────────┘
```

**事件命名空间**（`app/schema/chat_schema.py`）：每条 Instruction 有 `Header{type, namespace, name, ...}`，由 `ChatSpec.get_spec_name_info()` 从类限定名拆出。主要类：

- `UI.Status`：`phase` ∈ thinking/executing/retrying/finalizing
- `Template.TokenStream`：`token`、`is_final`
- `Template.ToastStream`：`stream`
- `Template.CallTool`：`id`、`service_name`、`tool_name`、`tool_params`、`friendly_name`
- `Template.CallToolResult`：`id`、`success`、`tool_name`、`tool_response`、`error_message`
- `Dialog.Exception` / `Dialog.Finish`

### 4.2 视觉推理流程

```
Camera (OpenCV, 独立线程)
    │
    ▼
CameraStream._worker (主循环, frame_interval_ms=1000 抓帧)
    │
    ├──→ MJPEG 编码 → /api/video_feed (浏览器预览)
    │
    └──→ MotionDetector.assess()  (每 motion_check_interval_seconds=0.2)
         │  比对当前帧 vs 上次推理的参考帧 (commit_reference 后更新)
         │
         ├─ distance ≤ motion_threshold(15) 且未超 max_idle_interval(60s) → 跳过
         │
         └─ 触发: 运动 ∨ 心跳超时, 且距上次推理 ≥ min_infer_interval(3s)
              │
              ▼
         run_coroutine_threadsafe(_run_inference_async, main_loop)  ← 跨线程到主事件循环
              │
              ▼
         VisionService.classify_frame(frames[:vision_use_img_count=3])
              │
              ▼
         VL 模型 → ActionResult(action, feedback, details)
              │
              ├─→ _resolve_display_result() (presence_threshold=3 平滑, 仅面板显示)
              │
              └─→ on_inference_done 回调 → AutomationAgent.trigger_evaluate() (call_soon_threadsafe)
```

### 4.3 自动化 / 定时任务

两条独立的执行路径，都最终落到 `ToolExecutor.execute_tool_by_name`：

```
[条件触发]  AutomationAgent._tick_loop (每 eval_interval=10s)
              │  或 on_inference_done 事件驱动 (call_soon_threadsafe)
              ▼
            AutomationService.evaluate(frames)
              │  ├─ 有 frames → vision_service.evaluate_condition (每 tick 编码一次, 多规则复用)
              │  └─ 无 frames → use_context_only (chat LLM 判时间/天气)
              ▼
            满足条件 → _run_actions → ToolExecutor → call_service 等

[定时触发]  SchedulerService._tick_loop (每 1s)
              │  找 enabled 且 next_run_at ≤ now 的任务
              ▼
            _execute_task
              │  ├─ payload.kind=tool  → ToolExecutor.execute_tool_by_name
              │  └─ payload.kind=message → Dispatcher.dispatch (非流式, 投到主会话)
              ▼
            at 任务执行后自动 enabled=false; every/cron 重算 next_run_at
```

## 6. 关键设计模式

### 6.1 ReAct Agent (LangGraph)

`build_chat_agent(tools)`（`app/agents/langgraph_agent.py`）：

- 用 `resolve_key_for_role("chat")` 拿 chat 角色 Key，建 `ChatOpenAI(streaming=True, temperature=0.7)`。
- `llm.bind_tools(tools)` 绑定所有 LangChain 转换后的 MCP 工具（用**短名**，`convert_all_tools(full_name=False)`）。
- `create_react_agent(llm, tools, post_model_hook=make_post_model_hook())`。
- `run_agent_streaming` 消费 `agent.astream_events(version="v2")`，120s 超时，把事件映射成 token/tool_start/tool_end/error。
- `post_model_hook` 在失败重试轮里剥掉已成功的 tool_calls（用 id 复用 AIMessage）。

### 6.2 MCP 工具层

- **内置工具**：`MCPClientManager.register_tool()` 注册，全名 `client_id___tool_name`（三下划线）。
- **外部工具**：stdio 启动子进程，JSON-RPC 2.0 通信。
- **LangChain 转换**：`convert_all_tools()` 转 `StructuredTool`，默认用短名。
- **运行时管理**：`/api/mcp/servers` 可连接/断开，但只能连 `external_mcp` 预声明的（白名单防 RCE）。
- **路由**：`ToolExecutor.resolve_tool_name()` 接受短名或全名，内部统一解析。

### 6.3 Motion-Gated Vision

- dHash（`hash_size=16` → 256 位指纹）做运动检测，微秒级。
- **参考帧策略**：和「上次推理帧」比（`commit_reference()` 在推理后才更新），不是上一帧——缓慢入画也能累积触发。
- 三道闸门：运动 `∨` 心跳超时，叠加 `min_infer_interval` 节流。
- `presence_threshold=3` 平滑**只影响面板显示**，自动化规则直接用 `evaluate_condition` 的 0/1，不走平滑。

### 6.4 动态实体控制

设备控件从 HA `attributes` + `services` 动态推导（`app/services/entity_controls.py`），零硬编码：

- **枚举控件**：数组属性 → 匹配 service field → 下拉
- **滑块控件**：数值属性 → min/max/step → 范围滑块（`_pct` 后缀优先）
- **动作控件**：无参 service → 概念匹配 → 按钮

### 6.5 多用户隔离

家庭局域网场景下，隔离粒度按"一家人"设计——会话和 LLM 配置各自独立即可，其余共享：

- JWT 登录后，`user_id` 通过 `dispatch_stream` → `SessionStore.get_or_create` 隔离会话（聊天记录互不可见）。
- `llm_keys`/`providers`/`home_info` 按 `user_id` 存在 `user_settings` 表（每人配自己的模型和密钥）。
- LLM Key 在运行时是**全局共享**的——用户改 Key 后 `reload_all_clients()` 重载全局客户端，不是请求级隔离。这是单实例家庭助手的合理简化，不追求多用户并发隔离。
- 定时任务、Emoji 偏好、天气查询地等不按用户隔离，一家人共享。
- 切换用户（`POST /api/users/switch`）重载 LLM 客户端 + 重写 `.env` + 重发 cookie。

### 6.6 热替换（`list[0]` 模式）

`ha_client_ref`、`automation_agent_ref`、`ha_catalog_cache_ref`、`ha_controls_cache_ref`、`scheduler_service_ref`（在 ToolDeps 里）、`dispatcher_ref`（在 SchedulerService 里）都是 `list` 单元素引用，运行时替换 `[0]` 即热替换，避免重建整个容器。

## 7. 服务初始化顺序

```
init_container(services, metrics_service)  (main.py:121)
  └─ AppContainer 字段就绪（含 ha_service/camera_stream/mcp_client_manager/tool_executor 等）

lifespan (main.py)
  ├─ Database.init()  (建表 + 迁移 user_id 列)
  ├─ emoji_service.load_index_async()
  ├─ rule_registry.load_from_db()
  ├─ session_store.load_from_db()
  ├─ 加载/迁移 vision_focuses (kv → vision_service)
  ├─ register_all_tools(deps)  (8 依赖工具 + 4 基础工具 = 12 内置)
  ├─ convert_all_tools → LangChain tools
  ├─ build_chat_agent(tools) → Dispatcher 创建
  ├─ AutomationAgent.start()  (eval_interval 从 automation.eval_interval_seconds 读)
  ├─ CameraStream.start()  (set_event_loop 注入主循环)
  ├─ SchedulerService.start()  (1s tick)
  ├─ 后台: _connect_external_mcp_servers()
  └─ 后台: _ha_catalog_refresh_loop() + 健康检查
```

> `dispatcher`、`scheduler_service`、`rag_service` 在 lifespan 里才赋值，之前为 `None`，路由里要判空。
