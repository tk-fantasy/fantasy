# 自动化引擎详解

> 对应代码：`app/services/automation_service.py`、`app/services/rule_service.py`、`app/services/rule_registry_service.py`、`app/agents/automation_agent.py`

自动化引擎实现从自然语言规则创建到条件评估与动作执行的完整闭环。与定时调度引擎互补：自动化处理**模糊条件**（视觉/时间/天气，靠 LLM 语义判断），定时任务处理**精确时刻**（at/every/cron，零判断）。

---

## 2. RuleService — 规则创建

`app/services/rule_service.py`

### 2.1 build_rule 流程

```python
async def build_rule(self, text: str) -> dict:
    # 1. 获取 HA 设备目录 + 服务定义 + 完整设备数据
    ha_catalog = self._ha_catalog_provider()
    devices = self._parse_ha_catalog(ha_catalog)
    full_devices = await self._ha_devices_provider()
    services_info = await self._ha_services_provider()

    # 2. 构建设备可控项中文文本（动态从 HA 属性+服务派生，零硬编码）
    #    链路: resolve_controls() → controls_to_text() → _filter_devices()
    controls_text = build_controls_text(full_devices, services_info)

    # 3. 构建系统提示词 → LLM 生成 JSON
    for attempt in range(MAX_RETRIES + 1):  # 最多 3 次
        content = await self._client.chat(messages, 20)
        parsed = self._parse_json(content)
        if not parsed:
            messages.append(assistant=content, user="JSON 解析失败，请重新生成")
            continue

        # 4. 校验 actions
        errors = self._validate_actions(parsed["actions"], full_devices, services_info)
        if not errors:
            return parsed

        # 5. 校验失败 → 带错误信息重试
        messages.append(assistant=content, user=f"请修正以下错误：\n{format_errors(errors)}")
```

**per-user 解析**（`rule_service.py:164-167`）：`build_rule` 开头先 `client = self._resolve_client(user_id)`，后续 LLM 调用用这个 client 而非 `self._client`。`_resolve_client`（`rule_service.py:51-72`）按 `user_id` 解析 chat role 的 per-user key（`resolve_key_for_role_user("chat", user_id)`），构造独立 `LlmChatClient(role="chat")` 并强制 `_enabled=True` 绕过全局开关；解析失败或 `user_id` 为空（老规则）回退全局 `self._client`。路由层 `rule_routes.py` 从 JWT 取 `user_id` 透传给 `build_rule` 和 `add_rule`。

### 2.2 action 校验规则

`_validate_actions()` 检查：

| 检查项 | 说明 |
|--------|------|
| `mcp_tool_name` 必须为 `ha_devices___call_service` | 唯一支持的设备控制工具（三下划线全名） |
| `domain` 存在 | 必须在 HA 服务定义中 |
| `service` 存在 | 必须在对应 domain 下 |
| `entity_id` 存在 | 必须在当前设备列表中 |
| `data` 字段合法 | 必须是该 service 需要的参数 |
| 枚举值合法 | 必须在可选值列表中（动态派生，非硬编码） |
| 设备-服务匹配 | service 字段必须与设备可控参数关联 |

设备可控项（枚举/滑杆/动作）由 `app/services/entity_controls.py` 的 `resolve_controls` 从 HA 实体属性 + 服务定义**动态派生**，零硬编码。详见《03-设备控制/控件类型》。

## 3. RuleRegistryService — 规则持久化

`app/services/rule_registry_service.py`

### 3.1 持久化机制

- **内存**: `threading.RLock` 保护的 Python list（评估热路径读内存，不读 DB）
- **持久化**: SQLite `rules` 表，整个 rule dict 存 JSON `data` 列
- **启动恢复**: `load_from_db()` 在 lifespan 中调用
- **CRUD**: `add_rule` / `set_enabled` / `delete_rule` / `update_trigger_time`

**`user_id` 字段**（创建者）：`AutomationRule.user_id: str = ""`（`rule_registry_service.py:32`），`to_dict()` 输出含此字段。`add_rule(rule, user_id)`（`:132-153`）优先用入参 user_id，路由层 `rule_routes.py` 从 JWT 注入。从 DB 加载时 `user_id=str(item.get("user_id", ""))`——老规则无此字段自然得 `""`，回退全局 chat key（保持原行为）。

```python
def add_rule(self, rule: dict) -> dict:
    # 1. 内存入队 (thread-safe)
    with self._lock:
        self._rules.append(normalized)
    # 2. 异步写 SQLite (fire-and-forget，不阻塞返回)

    return normalized.to_dict()
```

`update_trigger_time(rule_id, ts)` 仅更新内存 + 异步落盘 `last_triggered_at`，供冷却判断用。

## 4. AutomationAgent — 评估调度器

`app/agents/automation_agent.py`

### 4.1 双触发机制

```python
class AutomationAgent:
    async def _tick_loop(self):
        """定时评估主循环。每 eval_interval_seconds(默认10s) 一轮。"""
        while self._running:
            await asyncio.sleep(self._eval_interval)
            if now - self._last_eval_at >= self._eval_interval:   # 防抖
                await self._run_evaluation_cycle()

    def trigger_evaluate(self):
        """线程安全触发（从摄像头推理线程调用）。"""
        self._loop.call_soon_threadsafe(
            lambda: asyncio.ensure_future(self._run_evaluation_cycle())
        )

    async def _run_evaluation_cycle(self):
        self._eval_count += 1
        frames = await asyncio.to_thread(self._camera_stream.get_recent_frames)
        await self._automation_service.evaluate(frames=frames)
```

### 4.2 触发场景

| 场景 | 来源 | 线程 | 机制 |
|------|------|------|------|
| 定时 | `_tick_loop` 每 `eval_interval_seconds` | asyncio | `asyncio.sleep` + 防抖检查 |
| 推理完成 | 摄像头 `on_inference_done` 回调 | 摄像头推理线程 | `loop.call_soon_threadsafe` 跨线程投递 |

**跨线程关键点**：摄像头推理在独立线程跑，不能直接 `await` 主事件循环的协程，必须用 `call_soon_threadsafe` 把评估任务投递回 asyncio 主循环。`encode_frames_b64` 也是因此被设计成"一个 tick 编码一次复用"——避免每条规则在推理线程重复编码。

## 5. AutomationService — 规则评估

`app/services/automation_service.py`，核心常量 `_EVAL_TIMEOUT_SECONDS = 60`。

### 5.1 evaluate 流程

```python
async def evaluate(self, frames=None) -> list[dict]:
    rules = self._rule_registry.list_rules()
    vision_rules = []

    # 1. 过滤：禁用 / 冷却中 / 空条件 全部跳过
    for rule in rules:
        if not rule.get("enabled", True): continue
        if self._in_cooldown(rule, now): continue
        if not str(rule.get("condition", "")).strip(): continue
        vision_rules.append(rule)

    if not vision_rules:
        return applied

    # 2. 构建条件上下文（时间实时 + 天气60s缓存），所有规则共享
    context_info = await self._build_condition_context()

    # 3. 评估：两种模式都用 asyncio.gather 并行 + wait_for 60s 超时
    use_context_only = not frames or self._vision_service is None

    if use_context_only:
        eval_tasks = [self._evaluate_context_only(rule["condition"], context_info)
                      for rule in vision_rules]
    else:
        # 关键优化：一个 tick 编码一次，N 条规则复用同一份 b64
        pre_encoded_b64 = await self._vision_service.encode_frames_b64(frames)
        eval_tasks = [self._vision_service.evaluate_condition(
                          frames, rule["condition"],
                          context_info=context_info,
                          pre_encoded_b64=pre_encoded_b64)
                      for rule in vision_rules]

    try:
        results = await asyncio.wait_for(
            asyncio.gather(*eval_tasks, return_exceptions=True),
            timeout=_EVAL_TIMEOUT_SECONDS,        # 60s
        )
    except asyncio.TimeoutError:
        logger.warning("Rule evaluation timed out after %ds, skipping this cycle", 60)
        return applied                              # 整轮跳过，不执行任何动作

    # 4. 逐条处理结果：result==1 触发动作
    for rule, result in zip(vision_rules, results):
        get_container().metrics_service.record_automation_eval()
        if isinstance(result, Exception):
            logger.warning("Condition eval failed for '%s': %s", rule["name"], result)
            continue                                # 单条失败不影响其他规则
        if result == 1:
            applied.extend(await self._run_actions(rule, now))

    return applied
```

关键细节：

- **两种模式都并行**：context-only 和视觉模式都用 `asyncio.gather` 并行评估所有规则，不是串行。
- **整轮超时**：`asyncio.wait_for(..., 60)` 超时后整轮跳过（返回已收集的 `applied`，通常为空），防止单条规则卡死拖垮整个 tick。
- **单条容错**：`gather(..., return_exceptions=True)` 让单条规则的异常不传播，逐条 `isinstance(result, Exception)` 跳过。
- **encode_frames_b64 复用**：视觉模式下先编码一次 b64，作为 `pre_encoded_b64` 传给所有规则的 `evaluate_condition`，避免每条规则重复 `imencode + base64`（N 条规则省 N-1 次编码）。编码失败则 `pre_encoded_b64=None`，回退到每条规则各自编码。
- **metrics 集成**：每条规则评估记 `record_automation_eval()`，每次动作执行记 `record_tool_call(resolved)`，失败记 `record_tool_call(resolved, error=True)`。

### 5.2 条件上下文构建

```python
async def _build_condition_context(self):
    parts = []
    # 时间：零成本，每次实时获取
    time_data = await current_time_handler({"tz_offset_hours": _get_tz_offset_hours()}, None)
    parts.append(f"当前时间：{date} {weekday} {time}")

    # 天气：60s TTL 缓存，避免频繁请求外部 API
    if self._weather_cache is None or (now - self._weather_cache_at) >= 60:
        data = await get_weather_handler({}, None)
        if isinstance(data, dict) and "error" not in data:
            self._weather_cache = data
            self._weather_cache_at = now

    if self._weather_cache:
        parts.append(f"天气：{format_weather_brief(self._weather_cache)}")
    return "\n".join(parts)
```

- **时间**：用 MCP 工具 `current_time_handler`，按本机时区偏移算，零成本每次实时。
- **天气**：60s 缓存（`_weather_cache` + `_weather_cache_at`），失败静默降级（不阻塞评估，只是 context 少一行天气）。
- **上下文对所有规则共享**：一个 tick 只构建一次，传给所有 `evaluate_condition` / `_evaluate_context_only`。

### 5.3 动作执行

```python
async def _execute_action(self, task):
    # 兼容两套字段名：新 mcp_tool_* 优先，旧 tool_name/parameters 回退
    tool_name = task.get("mcp_tool_name") or task.get("tool_name")
    tool_input = task.get("mcp_tool_input") or task.get("parameters") or {}

    resolved = self._tool_executor.resolve_tool_name(str(tool_name))
    result = await self._tool_executor.execute_tool_by_name(resolved, dict(tool_input), None)

    get_container().metrics_service.record_tool_call(resolved)        # 成功计数
    if isinstance(result, dict) and result.get("success") is False:
        return None        # 工具返回失败：静默跳过（不抛异常）
    return {"tool": resolved, "result": result}
```

- **同链路**：`resolve_tool_name` → `execute_tool_by_name`，与定时调度引擎的 `tool` payload、与 WS 聊天的工具调用完全同出口。
- **静默失败**：工具返回 `success=False` 时返回 `None`（动作跳过），异常时也 `return None`（记 `record_tool_call(error=True)`），**不抛异常**——单条动作失败不影响同规则其他动作或同 tick 其他规则。
- **`resolve_tool_name`**：接受短名（`call_service`）或全名（`ha_devices___call_service`），统一解析。详见《MCP工具参考》。

### 5.4 纯上下文评估（_evaluate_context_only）

无摄像头帧时用 chat LLM 根据时间+天气判断，返回 0/1：

```python
async def _evaluate_context_only(self, condition, context, user_id):
    client = await self._resolve_chat_client(user_id)   # 按规则 user_id 解析
    prompt = f"当前环境信息：\n{context}\n\n请判断以下条件是否成立，只回复 1（成立）或 0（不成立）：\n{condition}"
    content = await client.chat([
        {"role": "system", "content": "你是一个条件判断器。只回复 1 或 0。"},
        {"role": "user", "content": prompt},
    ], 20)                                        # max_tokens=20
    m = re.search(r'\d+', content.strip())        # 抠第一个数字
    return 0 if int(m.group()) == 0 else 1        # 非0一律当1
```

- **按规则 user_id 解析 client**：`evaluate`（`automation_service.py:83-90`）把 `rule.get("user_id", "")` 透传给 `_evaluate_context_only`，**每条规则并发用自己的 user_id 解析**（不同用户创建的规则用各自的 chat key）。
- **`_resolve_chat_client(user_id)`**（`automation_service.py:221-247`）：`user_id` 非空时 `resolve_key_for_role_user("chat", user_id)` 构造独立 `LlmChatClient` + `_enabled=True`；为空（老规则）走 `else` 分支，懒加载一次全局 `self._chat_client` 复用（保持原行为）。
- **输出兜底**：用正则抠第一个数字，`0` 返回 `0`，其他任何数字一律当 `1`（容错模型胡言乱语）。
- **max_tokens=20**：只需一个字符，省 token。

## 6. Cooldown 机制

```python
def _in_cooldown(self, rule, now):
    cooldown = int(rule.get("cooldown_seconds", 10))       # 默认 10s
    last = float(rule.get("last_triggered_at", 0.0))
    return (now - last) < cooldown
```

触发后 `_run_actions` 调 `update_trigger_time(rule_id, time.time())` 记录时间戳，冷却期内跳过评估。`cooldown_seconds` 默认 10 秒，创建规则时由 LLM 根据语义设定（如"有人就开灯"给短冷却，"浇花"给长冷却）。

## 7. 条件类型与评估路径

| 条件类型 | 示例 | 有帧时评估路径 | 无帧时评估路径 |
|---------|------|--------------|--------------|
| 视觉类 | "画面中有人挥手" | VL 模型 `evaluate_condition` 返回 0/1 | `_evaluate_context_only`（无视觉信息，大概率返回 0） |
| 时间类 | "晚上6点以后" | VL 模型（带 context） | `_evaluate_context_only`（chat LLM 判时间） |
| 天气类 | "下雨的时候" | VL 模型（带 context） | `_evaluate_context_only`（chat LLM 判天气） |
| 设备状态类 | "客厅灯开着" | VL 模型（不能直接查 HA） | `_evaluate_context_only`（不能直接查 HA） |

**注意**：自动化评估的 context 只有时间+天气，**不包含设备状态**。"客厅灯开着"这类条件在自动化引擎里只能靠 LLM 猜，不可靠。设备状态查询走聊天流的 `verify_condition` 工具（实时查 HA），不进自动化评估。`evaluate_condition` 返回值的判定见《视觉子系统详解》——它从 VL 模型回复里用正则抠数字，0=不成立，非0=成立。

## 8. 纯上下文 vs 视觉评估

| 维度 | 纯上下文模式 (`use_context_only`) | 视觉模式 |
|------|-------------|---------|
| 触发条件 | `not frames or vision_service is None` | 有帧且 vision_service 可用 |
| 输入 | 时间 + 天气文本 | 多帧图像 + 上下文文本 |
| 模型 | Chat LLM (`_evaluate_context_only`) | VL 模型 (`evaluate_condition`) |
| b64 编码 | 不需要 | `encode_frames_b64` 一次，N 条规则复用 |
| 并行 | `asyncio.gather` 并行 | `asyncio.gather` 并行 |
| 适合条件 | 时间/天气条件 | 视觉/画面条件 |
| 评估客户端 | `_chat_client`（缓存复用） | `vision_service`（注入） |

两种模式共用同一套超时（60s）、容错（return_exceptions）、冷却、动作执行逻辑，只是评估函数不同。

### 与定时调度引擎的协作

两条路径互不感知，可以组合：

- **定时 + 视觉**：cron 任务每天 22:00 调 `call_service` 关灯（精确时刻）；同时配视觉规则"画面中有人"时开灯（模糊条件）。两者独立触发，互不干扰。
- **复用同链路**：自动化的 action 和定时任务的 `tool` payload 都走 `ToolExecutor.execute_tool_by_name`，工具名/入参格式/错误返回完全一致。
- **不共享状态**：自动化规则的 `last_triggered_at`/`cooldown_seconds` 和定时任务的 `last_status` 是独立的，不会互相影响冷却。

典型选择：需要"看到什么才做"用自动化规则；需要"到点就做"用定时任务（见《定时调度引擎详解》）。
