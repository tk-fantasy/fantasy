# 定时调度引擎详解

> 对应代码：`app/services/scheduler_service.py`、`app/services/schedule_parser_service.py`、`app/routes/scheduler_routes.py`、`app/schema/api_schemas.py`

定时调度引擎（SchedulerService）与自动化引擎（AutomationAgent）是互补的两条执行路径：

| 引擎 | 触发条件 | 判断方式 | LLM 开销 |
| --- | --- | --- | --- |
| 自动化引擎 | 模糊条件（视觉/时间/天气） | VL/chat LLM 语义判断，靠轮询 | 每次评估都调用 |
| 定时调度引擎 | 精确时刻（at/every/cron） | 零判断，事件驱动到点即触发 | 仅创建时翻译一次 |

定时任务的执行**不自带新出口**，而是复用已有两条出口：

- `payload.kind=tool` → `ToolExecutor.execute_tool_by_name`（与 `AutomationService._execute_action` 同链路）
- `payload.kind=message` → `Dispatcher.dispatch`（往主会话发一句话，与 WS 聊天同入口）

## 一、触发类型与计算

### 1.1 三种 schedule 形态

| kind | 含义 | 字段 | 示例 |
| --- | --- | --- | --- |
| `at` | 一次性未来时刻 | `at`：本地时间 ISO 字符串（`YYYY-MM-DDTHH:MM:SS`，不带时区） | `{"kind":"at","at":"2026-07-08T10:00:00"}` |
| `every` | 固定间隔 | `every_seconds`：正数秒 | `{"kind":"every","every_seconds":1800}` |
| `cron` | 标准 cron 表达式 | `expr`：5 字段（分 时 日 月 周），本地时间，不转 UTC | `{"kind":"cron","expr":"0 8 * * *"}` |

### 1.2 compute_next_run

`compute_next_run(schedule, now_ts)` 返回下一次触发的 Unix 时间戳（秒），无下次返回 `None`：

```python
if kind == "at":
    dt = datetime.fromisoformat(at_str.replace("Z", "+00:00"))
    ts = dt.timestamp()
    return ts if ts > now_ts else None          # 已过期 → 不触发

if kind == "every":
    return now_ts + every_seconds               # 永远是"现在 + 间隔"

if kind == "cron":
    now_dt = datetime.fromtimestamp(now_ts)
    nxt = croniter(expr, now_dt).get_next(datetime)
    return nxt.timestamp()
```

关键点：

- `at` 已过期返回 `None`——不会补触发错过的任务（崩溃恢复见 3.2）。
- `every` 每次执行完后重算 `now + interval`，所以"每 30 分钟"=两次执行间隔 30 分钟，而非"每个整点的 :00/:30"。
- `cron` 用 `croniter.get_next` 从当前时刻算下一次，校验合法性用 `croniter.is_valid`。

### 1.3 summarize_schedule

人类可读摘要，供前端展示与日志：

| schedule | 输出 |
| --- | --- |
| `at` | `于 2026-07-08T10:00:00 执行一次` |
| `every` ≥3600s | `每 1.0 小时` |
| `every` ≥60s | `每 30.0 分钟` |
| `every` <60s | `每 30 秒` |
| `cron` | `cron: 0 8 * * *` |

## 二、SchedulerService 生命周期

### 2.1 依赖注入

```python
SchedulerService(
    db,                 # Database — 持久化
    tool_executor,      # ToolExecutor — tool payload 出口
    dispatcher_ref,     # list[Dispatcher | None] — [0] 热替换模式
    session_store,      # SessionStore — 找主会话
    task_manager=None,  # 可选，用 TaskManager.spawn 派发执行
)
```

`dispatcher_ref` 用 `list[0]` 引用模式（与 `AppContainer.automation_agent_ref` 同模式），是为了支持运行时热替换 Dispatcher 而无需重启调度器。

### 2.2 start / stop

```python
async def start(self):
    await self._load_tasks()                                    # 从 DB 载入 + 恢复
    self._running = True
    self._tick_task = asyncio.create_task(self._tick_loop(),
                                          name="scheduler-tick")

async def stop(self):
    self._running = False
    self._tick_task.cancel()                                    # 中断 tick 循环
```

### 2.3 启动恢复（_load_tasks）

从 DB 载入全部任务到内存表 `self._tasks: dict[id, task]`（读多写少，CRUD 同步两边）。对每个启用任务：

1. 保留历史 `last_status`。
2. **崩溃恢复**：若 `last_status == "running"`（上次进程异常退出时遗留），改写为 `"interrupted"` 并填 `last_error="进程重启，上次执行被中断"`。
3. 重算 `next_run_at = compute_next_run(schedule, now)`。

禁用任务的 `next_run_at` 置 `None`。

## 三、tick 循环与执行

### 3.1 _tick_loop / _tick_once

```python
_TICK_INTERVAL_SECONDS = 1.0

async def _tick_loop(self):
    while self._running:
        try:
            await self._tick_once()
        except asyncio.CancelledError:
            raise
        except Exception:
            logger.exception("scheduler tick failed")      # 不让单次异常打断循环
        await asyncio.sleep(_TICK_INTERVAL_SECONDS)
```

`_tick_once` 单次扫描逻辑：

1. 遍历 `self._tasks`，跳过 `enabled=False` 和 `id in self._executing`（正在执行）的任务。
2. `next_run_at is not None and next_run_at <= now` → 加入 due 列表。
3. 每个 due 任务：`self._executing.add(id)` → `TaskManager.spawn(_execute_task)` 或裸 `asyncio.create_task`。

**精度**：1 秒 tick，所以 `at` 类型最坏延迟 1 秒。对本地助手够用；要更高精度需调小 `_TICK_INTERVAL_SECONDS`。

### 3.2 _execute_task（核心执行）

```python
async def _execute_task(self, task):
    task["last_status"] = "running"
    task["last_run_at"] = now
    await self._db.scheduled_task_update(task_id, task)

    try:
        if kind == "tool":
            await self._execute_tool_payload(payload)
        elif kind == "message":
            await self._execute_message_payload(payload, name)
        elif kind == "reminder":
            await self._execute_reminder_payload(payload, task.get("user_id", ""))
        else:
            raise ValueError(f"unknown payload kind: {kind}")
        task["last_status"] = "success"
    except Exception as exc:
        task["last_status"] = "failed"
        task["last_error"] = str(exc)
    finally:
        # at 类型一次性：成功/失败后禁用（不自动重试，下次到点自然不再触发）
        if schedule.get("kind") == "at":
            task["enabled"] = False
            task["next_run_at"] = None
        else:
            task["next_run_at"] = compute_next_run(schedule, time.time())
        await self._db.scheduled_task_update(task_id, task)
        self._executing.discard(task_id)
```

要点：

- **状态机**：`""` → `running` → `success` / `failed`；崩溃遗留 `running` → 启动恢复为 `interrupted`。
- **at 一次性语义**：无论成功还是失败，`finally` 里都 `enabled=False`。**不自动重试**——一次性任务失败就失败，需要手动重启用 `/run` 触发。
- **周期任务**：`finally` 里重算 `next_run_at`，所以即便执行耗时较长，下次触发也会顺延（不会"补触发"）。
- **重叠保护**：`_executing: set[str]` 防止 `every` 周期短于执行时长时并发执行同一任务（tick 跳过 `id in _executing` 的任务）。

### 3.3 tool payload 执行

```python
async def _execute_tool_payload(self, payload):
    tool_name = payload["tool_name"]            # 短名或全名都行
    tool_input = payload.get("tool_input") or {}
    resolved = self._tool_executor.resolve_tool_name(tool_name)
    result = await self._tool_executor.execute_tool_by_name(resolved, dict(tool_input), None)
    if isinstance(result, dict) and result.get("success") is False:
        raise RuntimeError(f"tool {resolved} returned failure: ...")
```

与 `AutomationService._execute_action` **完全同链路**：`resolve_tool_name` → `execute_tool_by_name`。区别仅是错误处理——调度器把工具返回的 `success=False` 当作任务失败（抛 `RuntimeError` 写进 `last_error`），而自动化引擎是静默跳过。

工具名接受短名（`call_service`）或全名（`ha_devices___call_service`，三下划线），由 `ToolExecutor.resolve_tool_name` 统一解析。详见《MCP工具参考》。

### 3.4 message payload 执行

```python
async def _execute_message_payload(self, payload, task_name):
    message = payload["message"].strip()
    dispatcher = self._dispatcher_ref[0]
    session_id = await self._resolve_main_session_id()     # 最近活跃会话
    if session_id is None:
        raise RuntimeError("无可用会话，无法投递定时消息")
    event = Event.build_event(Nlp.Request(query=message), request_id=rid, session_id=session_id)
    instructions = await dispatcher.dispatch(event)
```

- **不新建独立 agent 会话**——就是定时器替你"说一句话"进对话流，结果自然进会话历史（与 WS 聊天走同一入口 `Dispatcher.dispatch`）。
- **主会话选择**：`_resolve_main_session_id(task.user_id)` 取该用户最近活跃会话（`session_store.list_summaries(user_id)` 已按 `updated_at` 倒序，取第一个）。
- **无会话则失败**：用户从未创建过会话时，任务 `last_status=failed`、`last_error="无可用会话..."`。这是设计上的硬约束——定时消息必须有承接的会话。
- **无 `user_id` 直接拒绝**（`scheduler_service.py:303-305`）：message payload 缺 `user_id` 时抛 `RuntimeError` 不执行。这是防线——避免回退全局 agent（其 httpx 客户端可能被 per-user 构建误关，触发 Connection error）。

### 3.5 reminder payload 执行（独立链路）

```python
async def _execute_reminder_payload(self, payload, user_id):
    source = payload.get("original") or payload.get("intent")   # 优先用原话
    client = await self._resolve_reminder_client(user_id)        # 按 user_id 解析 chat client
    session_id = await self._resolve_main_session_id(user_id)
    # 组装：system + 会话历史 + directive，直接调 client.chat
    messages = [system_prompt, *session.model_messages, {"role":"user","content":directive}]
    reply = await client.chat(messages, timeout=60)
    # 把 directive + reply 追加进会话历史并持久化
```

**与 message 的区别**——reminder 是**绕开 ReAct 的独立链路**：

- 不调 `Dispatcher.dispatch`，不进 ReAct 循环。直接组装消息列表调 `client.chat()`，让 LLM 生成一句"主动开口"的提醒。
- **为什么独立**：message payload 把存死的文本当用户发言注入，AI 会走完整 ReAct——可能把"下班时间到了"又解析成新任务、调工具建提醒，形成副作用循环。reminder 直接告诉 LLM"这是定时触发的提醒，只输出一句话，不要调工具，不要建任务"。

**payload 字段**（`scheduler_service.py:326-387`）：

```jsonc
{ "kind": "reminder", "intent": "下班提醒", "original": "在18点27分提醒我下班" }
```

- `original`：用户创建任务时的原话，**优先使用**（语气更贴合用户语境）。
- `intent`：LLM 提炼的意图简述，作为兜底。
- 两者至少有一个，否则 `ValueError`。

**指令模板**（directive，拼在消息末尾）：

> [定时提醒触发] 用户创建此任务时说：「{source}」。现在是 {HH:MM}，已到预定提醒时刻。请你作为助手主动提醒用户——语气自然简短。**只输出提醒这一句话，不要调用任何工具，不要创建新任务。**

**`_resolve_reminder_client`**（`scheduler_service.py:389-410`）：按 `task.user_id` 解析 chat role 的 per-user key（与 message 同 role，避免 reminder 走 summary 模型导致语气不一致）。构造独立 `LlmChatClient(role="chat")`，覆盖 `_api_key/_base_url/_model`，并强制 `_enabled=True` 绕过全局 `llm.enabled` 开关（否则 `LlmBaseClient.post_json` 会拦"LLM 未启用"）。解析失败或无 per-user 配置回退全局 `self._llm_chat_client`。`user_id` 为空不会硬拒绝（与 message 不同），而是回退全局 client。

## 四、CRUD

写 DB + 更新内存 + 必要时重算 `next_run`：

| 方法 | 行为 |
| --- | --- |
| `add_task(task)` | 补 `id`(uuid)/`created_at`/`updated_at`/`last_status=""`/`last_run_at=None`/`next_run_at`/`user_id`(默认`""`，路由层注入创建者)，写 DB + 内存 |
| `update_task(id, patch)` | 合并 patch（不覆盖 `id`/`created_at`）；`schedule` 或 `enabled` 改动 → 重算 `next_run_at` |
| `delete_task(id)` | 内存 + DB 双删 |
| `set_enabled(id, enabled)` | 等价 `update_task(id, {"enabled": enabled})`，启用时重算 `next_run_at` |
| `run_now(id)` | 手动触发一次（不等 schedule），用 `_executing` 防重叠，正在执行则跳过返回当前 task |
| `list_tasks()` | 按 `created_at` 正序返回全部 |
| `get_task(id)` | 内存直读 |

### task dict 完整形态

```jsonc
{
  "id": "uuid-字符串",
  "name": "每天早上提醒起床",
  "enabled": true,
  "schedule": { "kind": "cron", "expr": "0 8 * * *" },
  "payload": { "kind": "reminder", "intent": "起床提醒", "original": "每天7点提醒我起床" },
  "user_id": "u_abc123",              // 创建者，路由层从 JWT 注入；空串=老任务
  "created_at": 1720000000000,        // 毫秒
  "updated_at": 1720000000000,
  "last_status": "",                  // "" | running | success | failed | interrupted
  "last_error": "",
  "last_run_at": null,                // 秒级时间戳
  "next_run_at": 17200000800.0        // 秒级时间戳，null=不触发
}
```

payload 三种形态：

```jsonc
// tool：调用 MCP 工具
{ "kind": "tool", "tool_name": "ha_devices___call_service", "tool_input": { "domain":"switch", "service":"turn_on", "entity_id":"switch.light" } }

// message：往主会话发一句话（走 ReAct，可能触发工具/建新任务）
{ "kind": "message", "message": "该起床了" }

// reminder：绕开 ReAct，直接生成一句提醒语（主动开口，不调工具）
{ "kind": "reminder", "intent": "下班提醒", "original": "在18点27分提醒我下班" }
```

## 五、持久化

### 5.1 表结构

```sql
CREATE TABLE IF NOT EXISTS scheduled_tasks (
    id TEXT PRIMARY KEY,
    data TEXT NOT NULL,           -- 整个 task dict 的 JSON
    created_at INTEGER NOT NULL,  -- 毫秒
    updated_at INTEGER NOT NULL
);
```

**家庭局域网单实例**，配置 + 运行时状态（`last_status`/`last_run_at`/`next_run_at`/`enabled`/`user_id`）合并存一个 JSON 列。量级小，不需要拆配置列/运行时列。

**`user_id` 在 data JSON 内**——任务带创建者字段，reminder/message 执行时按 `user_id` 解析对应 chat key（见 3.4/3.5）。**启动恢复兼容**：`_load_tasks`（`scheduler_service.py:159-170`）发现无 `user_id` 的旧任务时**禁用并标记**，避免老任务误执行（老任务在 per-user 化前的遗留，无创建者上下文）。

### 5.2 Database 方法

| 方法 | SQL |
| --- | --- |
| `scheduled_tasks_all()` | `SELECT id, data FROM scheduled_tasks ORDER BY created_at` |
| `scheduled_task_insert(id, data)` | `INSERT INTO scheduled_tasks (id, data, created_at, updated_at) VALUES (?,?,?,?)` |
| `scheduled_task_update(id, data)` | `UPDATE scheduled_tasks SET data=?, updated_at=? WHERE id=?` |
| `scheduled_task_delete(id)` | `DELETE FROM scheduled_tasks WHERE id=?` |

每次 tick 执行前后都会 `scheduled_task_update` 落盘运行时状态，崩溃后 `last_status` 不会丢（用于 3.2 的 `interrupted` 恢复）。

## 六、自然语言翻译（schedule_parser_service）

把"每天早上8点""明天10点""每30分钟"翻译成 `at`/`every`/`cron` 配置，**只在创建任务时调用一次**，运行时零 LLM 开销。

### 6.1 调用链

```
POST /api/scheduled-tasks/parse-schedule { phrase }
  → schedule_parser_service.parse_schedule(phrase)
      → _load_model_config_from_config()        # 复用 chat 角色配置
      → ChatOpenAI(model, temperature=0.0)      # 无工具的纯单次调用
      → ainvoke([("system", prompt), ("human", phrase)])
      → _extract_json(raw)                       # 容忍代码块标记/前后多余文本
      → if "error" in schedule: raise ValueError
      → _validate_schedule(schedule)             # at ISO / every 正数 / cron is_valid
      → return { "schedule": {...}, "summary": summarize_schedule(...) }
```

### 6.2 _SYSTEM_PROMPT 要点

- 强制**只输出 JSON 对象**，不要解释或代码块标记。
- 三类规则：单一未来时刻→`at`（本地时间 ISO，不带时区后缀）；纯间隔→`every`（整数秒）；周期时刻→`cron`（5 字段，本地时间，不转 UTC）。
- 给出 cron 示例对照：每天8点 `0 8 * * *`、工作日下午5点半 `30 17 * * 1-5`、每周一9点 `0 9 * * 1`、每月1号0点 `0 0 1 * *`。
- **当前时间注入**：prompt 里有 `{now}` 占位符，用 `str.replace` 注入（不能用 `str.format`，prompt 里有大量字面量 `{}` JSON 示例会被当字段解析）。
- 模糊描述兜底：返回 `{"error":"时间描述不明确"}`，上层抛 `ValueError` 提示用户。

### 6.3 _validate_schedule

翻译完成后强制校验，非法则抛 `ValueError` 让 API 返回友好错误：

| kind | 校验 |
| --- | --- |
| `at` | `at` 非空，`datetime.fromisoformat` 能解析（去掉 `Z`） |
| `every` | `every_seconds` 是正数 |
| `cron` | `expr` 非空且 `croniter.is_valid(expr)` |
| 其他 | `ValueError("未知触发类型")` |

### 6.4 _extract_json 容错

模型偶尔不遵守"只输出 JSON"约定，按顺序尝试：

1. `json.loads(text)` 直接解析整段。
2. 抠 ` ```json ... ``` ` 或 ` ``` ... ``` ` 代码块。
3. 抓第一个 `{...}` 子串。

三者都失败抛 `ValueError`。

## 七、API 路由

所有路由挂在 `/api` 前缀下，认证由全局中间件统一处理。

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/scheduled-tasks/parse-schedule` | 自然语言 → `{schedule, summary}` |
| GET | `/api/scheduled-tasks` | 列出全部任务（按 `created_at` 正序） |
| POST | `/api/scheduled-tasks` | 新建任务，body=`ScheduledTaskCreateRequest` |
| POST | `/api/scheduled-tasks/{id}/enabled` | 启停，body=`{enabled: bool}` |
| POST | `/api/scheduled-tasks/{id}/run` | 手动触发一次（调试用） |
| DELETE | `/api/scheduled-tasks/{id}` | 删除任务 |

`ScheduledTaskCreateRequest`：

```jsonc
{
  "name": "每天早上提醒起床",        // 必填
  "schedule": { "kind": "cron", "expr": "0 8 * * *" },   // 必填
  "payload": { "kind": "message", "message": "该起床了" }, // 必填
  "enabled": true                   // 默认 true
}
```

调度器未就绪时（`container.scheduler_service is None`）所有 CRUD 返回 `{"success":false,"message":"调度器未就绪"}`。

## 八、与自动化引擎的协作

两条路径互不感知，但可以组合使用：

- **定时 + 视觉**：用 cron 任务每天 22:00 调 `call_service` 关灯（精确时刻）；同时配一条视觉规则"画面中有人"时开灯（模糊条件）。两者独立触发，互不干扰。
- **复用同链路**：定时任务的 `tool` payload 和自动化的 action 都走 `ToolExecutor.execute_tool_by_name`，所以工具名、入参格式、错误返回完全一致——前端配置动作时可以共用同一套工具选择器。
- **不共享状态**：定时任务的 `last_status` 和自动化规则的 `last_triggered_at`/`cooldown_seconds` 是独立的，不会互相影响冷却。

典型选择：需要"到点就做"用定时任务；需要"看到什么才做"用自动化规则。
